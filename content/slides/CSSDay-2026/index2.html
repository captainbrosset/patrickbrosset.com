Example: XHR → Fetch

  • XMLHTTP: IE5 ActiveX experiment (2000), from the Outlook team
  • Used in weird ways → GMail, Web2.0, the AJAX era
  • Reverse-engineered by other browsers
  • Standardized as XMLHttpRequest
  • Much later: informed fetch()

XMLHTTP was originally an IE5 ActiveX experiment, in 2000, from the Outlook team.

Was used in weird ways, together with other APIs, making it possible to build GMail, and other Web2.0 apps. That gave us the AJAX era.

Later reverse-engineered and implemented by other browsers. Eventually standardized as XMLHttpRequest. Much later, it informed fetch().

Story shows that this obscure thing the Outlook team needed, then got used in a new way by other devs, leading to new use cases, which led to a standard, and then to a better standard.

The web doesn't improve by writing specs in a vacuum. The web improves because developers try new things, experiment, and complain loudly enough. Then, later, specs happen.

The wind tunnel analogy

  • Engineers don't standardize airplane designs before testing them
  • They build experimental prototypes
  • They shake them in wind tunnels
  • They discover turbulences
  • Then they write the rules for safe flight

If nobody is experimenting: new features don't get real usage patterns, edge-cases don't get discovered, ergonomics problems stay theoretical, interop issues never surface early.

Serious masonry

Let's dissect this new layout primitive.

Ok, let's dissect this new layout primitive, see how it works, and how you might use it in practice. I wasn't invited here just to do cool shit with CSS.

High-level overview

  • New name: grid-lanes
  • Not a special grid — fundamentally different
  • 1D, like flexbox (unlike grid)
  • Can be vertical or horizontal, and reversed
  • Lanes can be any size, uses repeat()
  • Items placed by the grid-lanes algorithm (opportunistic)
  • Can assign items to specific lanes, span multiple lanes
  • Visual order ≠ DOM order

There's a new name: grid-lanes, get used to it! It's not a special grid. It shares some things with CSS grid, but it's fundamentally different.

Grid-lanes is 1D, like flexbox, unlike grid. You don't get to control the other direction.

It isn't only vertical. It can be horizontal too, and reversed.

By default, you don't get to control the placement of items. They're placed according to the grid-lanes algorithm, which is opportunistic and sometimes can seem chaotic.

But you can still assign an item to a specific lane, and items can span multiple lanes.

Visual order is not DOM order. Items that appear later in the DOM may appear visually above earlier items. Keyboard tabbing order follows DOM order, not visual order.

Naming

  • "Masonry" — most common name, also a JS library
  • Mozilla's first prototype (~10 years ago): grid-template-columns: masonry;
  • Latest spec: display: grid-lanes
  • "Lanes" — swim lanes, driving lanes
  • Grid-lanes is 1D: you only control one direction

Masonry is the way most people have been referring to this, and it's still the most common name.

When the very first attempt was prototyped by Mozilla, almost 10 years ago, it was based off CSS Grid and used the "masonry" keyword value.

However, in the latest spec, the new name is grid-lanes, which is also a new display value.

The reason we use "lanes" over "columns and rows" like in grid, is that grid-lanes can be used in both directions, but you only get to control one direction.

Progressive enhancement

.layout {
  display: grid;
  display: grid-lanes; /* ignored by UAs without support */
}

@supports (display: grid-lanes) {
  .layout {
    display: grid-lanes;
  }
}

Grid and grid-lanes, for common use cases, can result in similar shapes. Grid-lanes is better at occupying space.

If you want to use it today, you can first declare a grid, and override with grid-lanes, which will be ignored by browsers that don't yet support this display type.

The @supports rule also works just fine for grid-lanes.

Shape and orientation

  • display: grid-lanes;
  • grid-template-columns → column/waterfall direction
  • grid-template-rows → row/brick wall direction
  • Full power of grid track sizing syntax
  • grid-column / grid-row for placement and spanning

In grid-lanes, you reuse some of the same properties as in grid. Set display:grid-lanes, then to set orientation, set either grid-template-columns (for columns/waterfall) or grid-template-rows (for rows/brick wall).

Again, masonry is 1D, unlike grid. In grid, you specify both axes. In masonry, you only define lanes along 1 dimension, and items get placed automatically.

You do have control over the shape and number of your lanes. grid-template-columns: 2fr 1fr 3fr 1fr for example. You can reuse the full power of the grid track sizing syntax.

Direction

  • Similar to flexbox: row/row-reverse, column/column-reverse
  • grid-lanes-direction: column track-reverse → fill from right-most lane
  • grid-lanes-direction: column fill-reverse → fill from bottom up

This is very similar to how flexbox works. In flexbox, you decide the shape and where the first item gets placed.

In masonry, at least in the current Chromium implementation, you can reverse things too.

Intrinsic lanes

grid-template-columns: 1fr 1fr;
grid-template-columns: repeat(3, 1fr);
grid-template-columns: 100px 200px auto 1fr;
grid-template-columns: repeat(auto-fill, minmax(100px, 1fr));

/* New in grid-lanes! */
grid-template-columns: repeat(auto-fill, auto);

You can use the full power of the grid track sizing syntax to define your lanes.

But there's one more thing you can now do with grid-lanes only: repeat(auto-fill, auto). Grid-lanes spec relaxes the auto-repeat syntax to allow intrinsic sizes like auto inside repeat(auto-fill, ...).

Flow

  • Same shape can be created with different layouts
  • It all depends on how items flow through the shape
  • Flexbox vs. multicolumn vs. grid-lanes
  • Masonry is opportunistic — fills the shortest lane first
  • Creates a criss-cross pattern

Beyond the general shape, it's very important to consider the order in which items are placed. The same general shape can be created with different layouts, and it depends on how you want your items to flow.

Masonry is opportunistic. It doesn't like to leave gaps. It'll try to fill the shortest lane first. And this is what gives us this criss-cross pattern.

Grid vs. grid-lanes

Grid gets you a grid, which can lead to gaps when items don't all have the same size.

It's easy to think grid-lanes is just grid with gaps squished together. But the placement algorithm is very different. Grid-lanes fills lanes one by one, always deciding which one is next. The layout is genuinely algorithmic.

Flow tolerance

  • Two conflicting goals: visual balance vs. preserve DOM order
  • Tolerance 0: always place in shortest lane
  • Default: 1em tolerance
  • Higher tolerance → more predictable lane order
  • infinite → always fill lanes sequentially

Masonry is trying to solve two conflicting goals: produce a visually balanced result, and preserve DOM order as much as possible.

At tolerance 0: the next item is always placed in the shortest lane. But there could be a tie, in which case the lane after the one last filled is picked.

Tolerance 0 is not the default because it can lead to visual jitter. The spec defines a 1em default tolerance.

At tolerance infinite: all tracks are always tied, so the algorithm always fills them one after the other. Helpful if you want to strictly follow lane order.
